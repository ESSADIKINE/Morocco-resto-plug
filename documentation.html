Vue d'ensemble du plugin Le Bon Resto

Introduction globale couvrant le positionnement, l'ADN et les objectifs du plugin de référencement gastronomique.

Le Bon Resto est un plugin WordPress verticalisé pour les annuaires gastronomiques qui combine une expérience frontend immersive avec un back-office structuré. Chaque composant a été pensé pour répondre aux besoins des offices du tourisme, des agences digitales et des groupes de restauration cherchant à industrialiser la publication de fiches restaurants à forte valeur éditoriale.

L'architecture repose sur un Custom Post Type "restaurant" extensible, des métadonnées spécialisées pour la localisation, l'offre culinaire et les médias enrichis, ainsi qu'un ensemble de shortcodes et de templates prêts à l'emploi. Grâce aux scripts Tailwind, Leaflet et Lightbox intégrés, la présentation conserve le style premium du thème tout en restant légère.

Ce guide décrit chaque fonctionnalité, les points d'extension et les meilleures pratiques pour orchestrer des campagnes multi-villes, automatiser les mises à jour et optimiser les performances SEO et Core Web Vitals sans surcharge.

  • Comprendre les briques fonctionnelles principales avant de configurer le plugin.
  • Identifier les rôles utilisateurs nécessaires : administrateurs, rédacteurs et opérateurs marketing.
  • Lister les sources de données (imports CSV, saisie manuelle, API tierces).
  • Préparer les ressources médias haute résolution pour les fiches établissements.
  • Planifier un calendrier éditorial pour les campagnes saisonnières et permanentes.

La documentation reprend cette charte : rubriques avec ruban coloré, cartes informatives arrondies et boutons accentués pour guider la navigation dans plus de 1000 lignes de contenu approfondi.

Déploiement

Installation, activation et premières vérifications

Procédure complète pour installer Le Bon Resto, vérifier l'environnement serveur et sécuriser l'activation.

Le plugin s'installe comme tout module WordPress : transférez le dossier "le-bon-resto" dans wp-content/plugins, ou utilisez le téléversement ZIP depuis l'interface d'administration. Lors de l'activation, le hook principal enregistre le Custom Post Type, initialise les options et affiche une notification de succès grâce au transient défini dans LeBonResto_Admin.

Aucune dépendance externe obligatoire n'est incluse, mais les fonctionnalités frontend reposent sur Tailwind CSS, Leaflet et Font Awesome chargés via CDN. Vérifiez que votre pare-feu autorise ces domaines ou préparez des copies locales en cas d'installation on-premise.

Après activation, ouvrez le menu "Le Bon Resto" dans l'administration pour accéder au tableau de bord, aux pages d'import/export et à l'écran de réglages. C'est également le bon moment pour permalinks -> "Enregistrer" afin de rafraîchir la réécriture d'URL du CPT.

  • Vérifier la version PHP (>=7.4) et la disponibilité des extensions cURL, JSON et mbstring.
  • S'assurer que l'utilisateur dispose des droits manage_options pour accéder aux sous-menus du plugin.
  • Confirmer que la réécriture d'URL est active pour servir les routes /restaurant/slug et /all.
  • Autoriser les requêtes sortantes vers les CDN utilisés pour Tailwind, Leaflet et Lightbox.
  • Créer un point de restauration de la base de données avant un import massif de restaurants.

Pré-requis serveur

Vérifications techniques indispensables avant de charger les données.

  • WordPress 6.x conseillé avec permaliens personnalisés pour une meilleure compatibilité SEO.
  • Mémoire PHP recommandée : 256 Mo pour supporter les imports et les visites simultanées.
  • HTTPS activé pour exploiter pleinement les micro-données et les appels API sécurisés.

Activation contrôlée

Points à surveiller lors de l'activation pour éviter les surprises.

  • Observation des notices d'administration générées par LeBonResto_Admin::admin_notices.
  • Confirmation de la présence du CPT "restaurant" via le menu latéral et l'écran Tous les restaurants.
  • Test rapide de la route REST /wp-json/lebonresto/v1/restaurants pour vérifier l'enregistrement de l'endpoint.

Checklist post-activation

Audit rapide immédiatement après l'installation.

  • Création d'une première fiche restaurant factice pour valider la saisie des métadonnées.
  • Aperçu du template /all pour s'assurer que Tailwind et Leaflet sont chargés correctement.
  • Contrôle du log serveur afin de repérer d'éventuels avertissements PHP ou erreurs 404 d'actifs.

Structure

Architecture des dossiers et organisation du code

Cartographie des répertoires, conventions de nommage et responsabilités de chaque module.

Le plugin est structuré autour de trois dossiers clés : assets/ pour les ressources frontales, includes/ pour les classes et fonctions PHP, et templates/ pour les vues frontales. Le fichier racine le-bon-resto.php orchestre l'initialisation et charge ces différents modules via les hooks WordPress.

Le dossier includes/ héberge les composants fonctionnels : enregistrement du CPT (cpt.php), endpoints REST (api.php), gestion des scripts (scripts.php), SEO (seo-meta.php, seo-advanced.php), optimisation (performance-optimization.php, html-optimization.php) et gabarits (templates.php). Cette séparation facilite la maintenance et les contributions multi-équipe.

Les assets CSS et JS sont spécialisés par page : all-restaurants.css/js pour le listing, restaurant-detail.css/js pour la fiche, single-restaurant.js pour l'expérience immersive et webp-support.js pour la compatibilité médias. Chaque fichier est appelé depuis les templates ou le gestionnaire de scripts centralisé.

  • Documenter les surcharges éventuelles si le thème enfant doit personnaliser un template.
  • Respecter la convention lebonresto_* pour les fonctions afin d'éviter les collisions.
  • Isoler les modifications spécifiques au client dans un mu-plugin si nécessaire.
  • Suivre les mises à jour Git afin de garder une trace claire des évolutions.
  • Préparer un schéma d'architecture pour les ateliers de transfert de compétences.

Répartition des rôles

Vue synthétique des dossiers clés et de leur mission.

  • assets/css : feuilles de style modulaires alignées sur la charte Le Bon Resto.
  • assets/js : scripts responsables des interactions (filtres, cartes, galeries, lazy loading).
  • includes : logique métier, API, SEO, optimisation et intégration WordPress.

le-bon-resto.php agit comme bootstrap du plugin.

  • Chargement conditionnel des classes et fichiers selon le contexte (admin ou public).
  • Enregistrement des hooks d'activation/désactivation pour nettoyer les transients.

Templates intégrés

Vue des fichiers de gabarits fournis nativement.

  • templates/all-restaurants.php : page annuaire avec filtres, cartes et infinite loading.
  • templates/single-restaurant.php : fiche immersive avec storytelling, carte et réservations.
  • templates/restaurant-detail.php : layout détaillé orienté storytelling et galerie médias.

Custom Post Type "restaurant" et taxonomies associées

Description détaillée du type de contenu, de ses capacités et des extensions possibles.

Le fichier includes/cpt.php enregistre le Custom Post Type "restaurant" avec un set complet de labels en français pour l'administration. Le CPT est public, consultable via /restaurant/{slug}, exposé dans l'API REST et attaché au menu "Le Bon Resto" afin de centraliser l'expérience de gestion.

Les capacités reposent sur le modèle standard des posts, ce qui permet de tirer parti du système de rôles natif de WordPress. Les supports activés incluent le titre, l'éditeur, l'image à la une, l'extrait et les champs personnalisés, indispensables pour injecter du storytelling et des descriptions SEO.

Aucune taxonomy n'est imposée par défaut, mais les métadonnées gèrent les notions de ville, type de cuisine, budgets et statut premium. Les projets nécessitant des taxonomies hiérarchiques peuvent les ajouter via register_taxonomy en conservant le namespace lebonresto_.

  • Définir les rôles utilisateurs et capabilities si des restrictions fines sont nécessaires.
  • Planifier un mapping avec d'éventuelles taxonomies existantes dans le thème ou un plugin tiers.
  • Standardiser les slugs des établissements pour garantir des URLs stables et SEO-friendly.
  • Vérifier que la réécriture /restaurant fonctionne sur tous les environnements (staging, prod).
  • Utiliser les templates single-restaurant.php et archive all pour contrôler le rendu du CPT.

Champs natifs

Ce que supporte le CPT sans extension.

  • Titre pour le nom commercial de l'établissement.
  • Éditeur WordPress pour la présentation narrative longue.
  • Image mise en avant pour alimenter les vignettes et hero banners.

Options d'affichage

Configuration de l'administration via manage_restaurant_posts_columns.

  • Colonnes personnalisées pour afficher ville, type de cuisine et statut mis en avant.
  • Tri possible par ville ou statut premium grâce à restaurant_sortable_columns.
  • Recherche améliorée sur les métadonnées via restaurant_search_where.

Hooks essentiels

Points d'extension principaux liés au CPT.

  • add_action('init', 'lebonresto_register_restaurant_cpt', 5) pour un chargement précoce.
  • add_action('add_meta_boxes', 'lebonresto_add_restaurant_meta_boxes') pour enrichir les formulaires.
  • add_action('save_post_restaurant', 'lebonresto_save_restaurant_meta') pour persister les données personnalisées.

Méta-boxes et champs personnalisés disponibles

Catalogue complet des champs éditoriaux accessibles depuis l'administration.

Six méta-boxes structurent la saisie des fiches : détails, médias, blog, options, menus et avis. Chacune utilise une nonce dédiée pour sécuriser la sauvegarde et mappe les données vers des clés meta préfixées par _restaurant_. Les valeurs sont ensuite exposées dans l'API REST grâce à lebonresto_add_restaurant_fields_to_rest.

La méta-box Détails rassemble les attributs principaux : description courte, adresse, ville, latitude/longitude, lien Google Maps, type de cuisine, téléphone, e-mail, fourchette de prix et devise. Un switch "Mis en avant" active le statut premium exploité sur la page d'accueil.

Les méta-boxes Médias et Blog enrichissent le storytelling avec une image principale, une galerie, une vidéo, une visite virtuelle 360°, un titre d'article et un contenu narratif. Les options multi-sélection permettent d'afficher les services proposés (wifi, terrasse, options végétariennes) tandis que la méta-box Menus accepte un JSON sérialisé ou un tableau via le builder intégré.

  • Former les contributeurs sur la saisie de latitudes/longitudes pour garantir l'affichage correct sur Leaflet.
  • Standardiser les libellés des types de cuisines pour simplifier les filtres frontend.
  • Valider la taille et le poids des médias afin d'éviter les ralentissements sur mobile.
  • Mettre en place une charte de rédaction pour le contenu blog associé à chaque fiche.
  • Contrôler régulièrement la cohérence des menus JSON via un export API.

Gestion des médias immersifs et galeries

Comment Le Bon Resto orchestre images, vidéos, visites virtuelles et galeries filtrables.

La méta-box Médias permet de déclarer une image principale, une galerie multi-images et des contenus interactifs (vidéos, visites virtuelles). Ces éléments sont exploités dans templates/restaurant-detail.php et single-restaurant.php via Lightbox et des conteneurs responsives pour les iframes.

Le script assets/js/restaurant-detail.js pilote l'affichage dynamique de la galerie, l'ouverture Lightbox, la gestion des onglets et l'affichage conditionnel des sections médias. Il s'assure que les éléments ne chargent qu'en présence de données pour préserver la performance.

Pour les besoins d'optimisation, le plugin propose assets/js/lazy-loading.js et assets/js/webp-support.js afin de vérifier la compatibilité WebP et différer le chargement des images hors écran. Ces modules peuvent être activés depuis includes/scripts.php ou intégrés dans un bundle personnalisé.

  • Préparer les médias en double format (JPEG + WebP) pour profiter de la détection automatique.
  • Limiter la taille des vidéos embarquées ou préférer l'intégration YouTube/Vimeo avec paramétrage privacy.
  • Renseigner les balises alt pour chaque image afin d'améliorer l'accessibilité et le référencement.
  • Tester l'ouverture Lightbox sur mobile pour vérifier la fluidité de la navigation tactile.
  • Segmenter les galeries par ambiance (salle, plats, équipe) pour faciliter la découverte.

Cartographie, géolocalisation et expérience Leaflet

Configuration du moteur cartographique et bonnes pratiques d'intégration.

Le plugin exploite Leaflet via les CDN officiels pour offrir une carte interactive légère et personnalisable. Les scripts assets/js/single-restaurant-map.js et assets/js/all-restaurants.js orchestrent l'affichage des marqueurs, des popups et la synchronisation avec les filtres.

Chaque restaurant stocke latitude, longitude et lien Google Maps. Les templates convertissent ces données en marqueurs Leaflet avec un style cohérent (icônes jaunes et popups sombres pour rappeler la charte graphique). Les popups incluent miniatures, note moyenne et lien vers la fiche détaillée.

Pour améliorer la pertinence des recherches, les utilisateurs peuvent saisir leur ville ou autoriser la géolocalisation navigateur. Le script détecte ensuite les établissements dans un rayon configurable via les options du plugin et affiche les distances dans la liste.

  • Saisir les coordonnées GPS avec précision (format décimal) pour éviter les décalages d'affichage.
  • Paramétrer le zoom par défaut dans les shortcodes [lebonresto_map_page] et [lebonresto_map_only] selon le territoire.
  • Prévoir des clusters si le catalogue dépasse plusieurs centaines de restaurants concentrés dans une ville.
  • Tester la carte sur terminaux tactiles pour vérifier le confort de navigation et les contrôles zoom.
  • Configurer un fallback visuel si Leaflet ne peut pas se charger (image statique ou liste).

  • lebonresto_map_page shortcode : expose une carte en pleine page pour les campagnes thématiques.

Filtres dynamiques, recherche et pagination

Description du moteur de filtrage côté frontend et des options disponibles côté API.

La page annuaire (templates/all-restaurants.php) combine une barre de recherche, des filtres par ville, cuisine, prix, statut premium et distance. Le JavaScript assets/js/all-restaurants.js interroge l'API REST /lebonresto/v1/restaurants en injectant les paramètres sélectionnés et affiche les résultats sous forme de cartes réactives.

L'API accepte des paramètres name, city, cuisine, distance, lat, lng, sort, featured_only, page, per_page, min_price et max_price. Les réponses sont mises en cache via des transients pour accélérer les requêtes répétitives. La pagination est gérée en interne et peut être configurée (per_page) via le shortcode [lebonresto_all_page].

Les templates intègrent également un système de skeleton loading et de messages d'état (chargement, aucun résultat, erreur). Les boutons "Afficher plus" et "Effacer les filtres" améliorent le parcours utilisateur et incitent à la découverte.

  • Limiter le nombre de filtres affichés simultanément sur mobile pour préserver l'ergonomie.
  • Prévoir des valeurs par défaut cohérentes (rayon, tri) dans les options du plugin.
  • Nettoyer les entrées utilisateur côté API avec sanitize_text_field et absint (déjà implémenté).
  • Associer un plan de test QA pour valider les combinaisons de filtres et les réponses JSON.
  • Monitorer les performances de l'endpoint via Query Monitor ou New Relic.

Paramètres REST

Résumé des arguments acceptés par l'API.

  • name, city, cuisine pour les filtres textuels.
  • lat, lng, distance pour la recherche géolocalisée.
  • page, per_page, sort pour contrôler la pagination et l'ordre d'affichage.

Expérience utilisateur

Composants UI clés du listing.

  • Chips filtrables et boutons radiaux pour les options populaires (terrasse, livraison, premium).
  • Affichage de badges mis en avant lorsque _restaurant_is_featured est activé.
  • Messages de fallback traduits via les strings localisés (loading, noResults, error).

Optimisations techniques

Gains de performance intégrés.

  • Transients lebonresto_restaurants_* purgés à chaque sauvegarde de restaurant.
  • Limitation du per_page à 100 pour éviter les requêtes trop lourdes.
  • Debounce des entrées utilisateur pour réduire les appels API inutiles.

Back-office

Interface d'administration et gestion des réglages

L'endpoint GET /wp-json/lebonresto/v1/restaurants renvoie un flux JSON de restaurants avec support de pagination, filtres avancés et données géolocalisées. Il répond true par défaut (public) pour permettre des intégrations front et externes.

Les paramètres acceptés couvrent la recherche textuelle (name), la ville (city), la cuisine (cuisine), le budget (min_price, max_price), le tri (sort=featured|newest|distance), la géolocalisation (lat, lng, distance) et la pagination (page, per_page).

Un cache transients stocke les réponses en fonction des paramètres pour réduire les requêtes répétées. Les transients sont purgés automatiquement lors de save_post_restaurant, delete_post et wp_trash_post.

  • Limiter per_page aux valeurs recommandées pour éviter les charges serveur.
  • Authentifier les appels lorsqu'une extension externe souhaite accéder à des données non publiques.
  • Mettre en place une surveillance du temps de réponse via un outil APM.
  • Utiliser l'endpoint pour alimenter des applications mobiles ou bornes interactives.
  • Documenter les codes d'erreur et messages renvoyés pour faciliter l'intégration.

Structure de réponse

Champs fournis par défaut.

  • post_title, permalink, excerpt.
  • restaurant_meta : description, adresse, ville, lat/lng, cuisine, prix, options.
  • distance calculée lorsque lat/lng utilisateur sont fournis.

Sécurité

Mesures incluses.

  • Sanitization des paramètres via sanitize_text_field, absint, floatval.
  • Nonce généré par wp_create_nonce pour les appels Ajax internes.
  • Limitation du temps d'exécution à 30s pour éviter les abus.
